# Linsight Demo 体验指南

> 本指南面向 TRAE AI 创造力大赛评委。照此操作，5–8 分钟可完整体验 Linsight 餐饮小店案例的全链路闭环。

---

## 一、环境要求

- **Node.js 20+**（推荐 22/24，后端使用 `node:sqlite` 内置模块，免 native 编译）
- **浏览器**：Chrome / Edge（推荐）
- **4 个终端窗口**：分别跑后端 + 三端前端
- **操作系统**：Windows / macOS / Linux 均可

---

## 二、启动步骤

### 1. 安装依赖（首次运行需要）

```bash
cd d:/Projects/Linsight/server  && npm install
cd d:/Projects/Linsight/web     && npm install
cd d:/Projects/Linsight/wechat  && npm install
cd d:/Projects/Linsight/h5      && npm install
```

### 2. 初始化数据库

```bash
cd d:/Projects/Linsight/server
npm run db:init   # 创建 SQLite 库 + 16 张表
npm run db:seed   # 填充 Demo 数据：3 供应商 / 30 食材 / 20 菜品 / 5 桌台 / 7 采购单 / 19 订单（覆盖 2026-07-03 ~ 07-09）
```

> 数据库文件生成在 `server/../linsight.db`。重置只需再跑一次这两条命令。

### 3. 启动 4 个服务（4 个终端）

**终端 1 — 后端 API（端口 3001）**
```bash
cd d:/Projects/Linsight/server
npm run dev
```
启动后访问 http://localhost:3001/api/health ，应返回 `{"status":"ok",...}`。

**终端 2 — 商家管理后台（端口 5173）**
```bash
cd d:/Projects/Linsight/web
npm run dev
```
访问 http://localhost:5173

**终端 3 — 老板微信端（端口 5174）**
```bash
cd d:/Projects/Linsight/wechat
npm run dev
```
访问 http://localhost:5174

**终端 4 — 顾客点餐 H5（端口 5175）**
```bash
cd d:/Projects/Linsight/h5
npm run dev
```
访问 http://localhost:5175/?table_no=A1

### 4. 打开落地页

双击 `d:/Projects/Linsight/index.html`，或用浏览器打开本地文件路径。

## 三、推荐体验路径（5–8 分钟闭环）

路径设计原则：**先看故事 → 看一天 → 亲自触发一个独占壁垒 → 看数据流转 → 看日报闭环**。

### 步骤 1｜落地页：理解产品与三个独占壁垒（1 分钟）

打开 `index.html`，依次滚动：

1. **Hero 区**：品牌「Linsight」+ 定位「AI 经营分析助手」
2. **三端入口卡片**：商家端 / 老板端 / 顾客端
3. **三个独占壁垒**（核心差异点，务必停留）：
   - 📷 **拍照入账**：老板拍照发微信，AI 自动识别进货单并入库
   - ⚠️ **损耗监控**：理论消耗 vs 实际消耗，差值即洞察
   - 📊 **日报推送**：早 8 点 AI 主动推送经营日报到微信
4. **老板的一天 · 时间轴控制器**（⭐ 重点交互）：点击时间轴节点（10:30 拍照入账 / 22:30 损耗监控 / 次日早 8 点），右侧场景卡片会切换内容，体验「老板的一天如何跑在 Linsight 上」。评委无需等一天，1 分钟看完闭环。
5. **餐饮小店工作流对比**：改造前（凭感觉、对不上账）vs 改造后（AI 算差异、主动告警）
6. **9 天冲刺时间轴**：Day1-3 已完成（绿）/ Day4 进行中（金）

### 步骤 2｜老板端：发图识别进货单 → 确认入库（1.5 分钟，独占壁垒①）

访问 http://localhost:5174 （默认进聊天页）

1. 点击底部输入栏右侧的「**＋**」按钮
2. 弹出底部 sheet，含 3 个预设进货单文本，点击第一个「张记生鲜：牛肉 15斤 42元/斤…」
3. AI 自动识别并返回**进货单预览卡片**：含商品明细（牛肉/青菜/鸡蛋 + 单价 + 置信度）、异常预警（牛肉单价偏高 8%、鸡蛋置信度低）、合计金额
4. 点击卡片底部「**确认入库**」按钮 → 卡片变为「✅ 进货单已确认入库，库存已更新」

> 这一步对应独占壁垒①：老板只做了一个本来就在做的动作（拍照发微信群），AI 完成了识别、入库、库存联动。

### 步骤 3｜商家端：看库存联动变化（1 分钟）

访问 http://localhost:5173/inventory （库存查看页）

1. 可见 30 个食材的当前库存、单位、预警阈值、是否低库存
2. 刚才确认入库的牛肉 / 青菜 / 鸡蛋库存已增加（步骤 2 的结果）

切到 http://localhost:5173/purchases （采购进货页）可看到刚才那张已确认的采购单。

### 步骤 4｜顾客端：下单触发数据产生（1 分钟）

访问 http://localhost:5175/?table_no=A1

1. 浏览菜品分类（主食 / 小菜 / 饮品），点击「+」加 2–3 道菜到购物车
2. 点击底部购物车栏展开明细 →「去下单」
3. 确认页填写备注 / 人数 →「确认下单」→ 跳转成功页（订单号 + 已通知厨房）
4. 进入支付页选支付方式 → 支付成功 → 订单状态页

> 这一步对应核心理念：数据在老板和员工本来就在做的动作里自然产生，不需要任何人「使用系统」。

### 步骤 5｜老板端：看经营日报与 AI 建议（1.5 分钟，独占壁垒③）

访问 http://localhost:5174/#/reports （日报 tab）

1. 选择日期（推荐 **2026-07-09**，seed 数据最完整）
2. 点击「生成日报」
3. 可见完整经营数据：营收 / 食材成本 / 人工成本 / 损耗 / 总成本 / 毛利 / 毛利率 / 净利 / 客流 / 客单价 / TOP5 菜品 / 单品利润明细
4. 可见 AI 摘要 + 行动建议（如「对账差异 -2132 元，建议核查」）

切到 http://localhost:5174/#/insights （洞察 tab）可看到 AI 主动产生的异常预警（损耗告警 / 对账差异 / 日报推送记录）。

> 这一步对应独占壁垒③：AI 主动把经营状况推到老板面前，老板不用打开任何报表。

### 步骤 6｜回到落地页：时间轴闭环（30 秒）

回到 `index.html` 的「老板的一天」时间轴，确认完整闭环：
早 8 点看日报 → 10:30 拍照入账 → 营业中下单 → 22:30 损耗监控 → 次日早 8 点新日报。

## 五、技术亮点

1. **合同先行**：`CONTRACT.md` 定义 16 张表、50+ API 端点，作为单一事实源，支撑 4 路并行开发
2. **三端联动**：顾客下单 → 商家收银 → 库存 FIFO 扣减 → 损耗对账 → 日报生成 → 微信推送，完整闭环
3. **AI 统一封装**：`callLlm()` 统一入口，无 key 时自动 fallback 到规则匹配，不阻断业务
4. **时间轴控制器**：落地页可交互场景切换，评委无需等一天即可体验「老板的一天」

## 六、常见问题

**Q：为什么 AI 回复像是规则匹配？**
A：当前未配置 `TRAE_API_KEY`，3 个 AI 端点（对话/日报摘要/进货单识别）自动走 fallback 规则匹配，结构完整但不调用真实大模型。配置 key 后即启用真实 AI（见 `.env.example`）。

**Q：为什么有的日报净利为正、有的为负？**
A：Demo 数据覆盖 7 天真实流水，不同日期营收与成本不同。AI 会根据数据主动提示「亏损」或「良好」，正好展示产品「帮老板发现问题」的价值。

**Q：数据库怎么重置？**
A：`cd server && npm run db:init && npm run db:seed` 即可重置到初始状态。

**Q：端口被占用怎么办？**
A：检查是否有其他服务占用 3001 / 5173 / 5174 / 5175，或修改对应目录下 `vite.config.ts` 与 `server/src/index.ts` 的端口配置。

**Q：顾客端下单后库存为什么有的食材没变？**
A：下单时按菜品配方（BOM）FIFO 扣减库存。若某食材当前无 active 库存批次（已消耗完），扣减会静默跳过——这是 Demo 数据的真实状态，不影响业务流程。
