# Linsight 评分维度对照

根据大赛评审的 4 个维度（创新性 30%、实用性 30%、完成度 20%、美观度/设计体验 20%），逐项说明 Linsight 的对应表现。所有功能均以实际 Demo 验证为准，不夸大。

---

## 一、创新性（30%）

### 1.1 需求创新

**评审标准**：是否精准识别了真实场景中，长期存在但尚未被有效解决的用户需求

**Linsight 的表现**：
- **真实痛点**：中国数以千万计的餐饮小店老板，请不起经营分析师，看不懂复杂报表，但每天都需要做经营决策
- **长期存在**：这个问题存在了 20 年，从纸质账本到 Excel 到 SaaS，都没有真正解决——因为工具太复杂，老板学不会
- **未被有效解决**：现有方案要么太重（BI 工具），要么太轻（记账 App），没有「嵌入工作流的 AI 分析」

**证据**：
- 落地页「痛点 × 解法」section 明确列出对比
- 创始人为商业分析背景，真实理解这个需求

### 1.2 解决思路

**评审标准**：解决方案是否具有突破性，是否以不同于现有产品的方式回应了该需求

**Linsight 的突破——三个独占壁垒**（本次 Demo 已全部跑通）：

| 壁垒 | 现有方案 | Linsight 的解法 | Demo 验证 |
|------|---------|----------------|----------|
| ① 拍照入账 | SaaS 要老板手动录入采购 | 老板发图 → AI 识别 → 确认入库 → 库存联动 | ✅ 老板端发预设进货单 → 收到识别卡片（含异常预警）→ 点确认入库 → 商家端库存增加 |
| ② 损耗监控 | 记账 App 只记不分析 | 理论消耗（销量×BOM）vs 实际消耗（期初+采购-期末），差值即洞察 | ✅ POST /loss-records/generate 返回 30 食材损耗 + AI 分析；盘点页录入实盘 → 差异告警 |
| ③ 日报推送 | BI 要老板主动打开 | AI 自动算指标 + 生成人话摘要 + 主动推送到微信 | ✅ POST /daily-reports/generate 返回完整指标；POST /push → 微信端收到日报卡片 |

**对比现有产品**：
| 产品 | 问题 | Linsight 的解法 |
|------|------|----------------|
| 传统 BI | 太重，老板学不会 | 人话洞察 + 行动建议 + 主动推送 |
| 记账 App | 只记录不分析 | AI 自动算指标 + 主动推送 |
| 问数机器人 | 需要知道问什么 | AI 主动发现问题（损耗/对账差异） |
| SaaS 系统 | 数据孤岛 + 要学新工具 | 三端联动 + 嵌入老板本就在做的动作 |

### 1.3 技术创新

**评审标准**：是否在 TRAE 能力运用上展现出独特思路

**Linsight 的技术创新**：
- **合同先行 + 并行开发**：用 CONTRACT.md 作为单一事实源，4 个 TRAE 会话并行开发，效率 4 倍；冲刺阶段扩展到 10 窗口并行修复
- **AI 统一封装**：`callLlm()` 接口，失败自动 fallback 到规则匹配，不阻断业务
- **工作流嵌入**：不是调用 AI API 那么简单，而是让 AI 嵌入整个经营流程（拍照→识别→入库→扣减→对账→损耗→日报→推送）
- **时间轴控制器**：落地页可交互场景切换，评委无需等一天即可体验「老板的一天」闭环

**TRAE 能力运用**：
- 用 TRAE 做架构设计与契约定义（Day 1）
- 用 TRAE 4 路并行开发后端/前端（Day 2-3）
- 用 TRAE model API 做 AI 对话/日报生成/OCR 识别（Day 3）
- 用 TRAE 10 窗口并行冲刺修复（Day 8）
- 用 TRAE 做全链路联调测试（Day 8）

## 二、实用性（30%）

### 2.1 场景洞察

**评审标准**：精准识别真实场景中的用户需求，目标用户群体明确

**Linsight 的场景**：
- **目标用户**：餐饮小店老板（1-5 张桌台，月营收 5-20 万）
- **真实需求**（Demo 均已覆盖）：
  - 今天赚没赚钱？→ 日报（营收/成本/毛利/净利）
  - 哪道菜最赚钱？→ TOP5 菜品 + 单品利润明细
  - 哪个食材快没了？→ 库存查看 + 低库存预警
  - 进货单怎么记？→ 拍照识别入库
  - 损耗有没有异常？→ 损耗监控 + AI 告警
  - 实盘对不对得上？→ 实盘录入 + 差异对账

**证据**：
- 16 张数据库表覆盖完整业务流程
- 50+ API 端点覆盖所有操作
- Demo 数据模拟真实小店 7 天流水（7-03 ~ 7-09）

### 2.2 解决效果

**评审标准**：能有效提升效率、降低成本、改善体验，用户有明确使用意愿

**Linsight 的效果**：
- **效率提升**：原来需要分析师花半天算的指标，现在 AI 自动完成并主动推送
- **成本降低**：不需要请经营分析师（月薪 8000+），AI 助手成本接近 0
- **体验改善**：从「我来看数据」到「数据来找我」，老板不费劲

**使用意愿**：
- 老板每天都在用微信，AI 助手在微信里，学习成本为 0
- 日报主动推送，不需要老板主动操作
- 拍照入库，不改老板任何习惯
- 人话建议，一看就知道该做什么

### 2.3 落地可行性

**评审标准**：当前 Demo 已能验证核心价值，而非纯概念

**Linsight 的落地**（9 步全链路联调验证通过）：
- ✅ **采购进货**：发图 → AI 识别 → 返回卡片含 order_id
- ✅ **确认入库**：confirm → status=confirmed → 库存批次插入
- ✅ **顾客下单**：POST orders → BOM 库存 FIFO 扣减
- ✅ **支付**：checkout → status=paid + 支付记录
- ✅ **库存联动**：牛肉 25→24.7kg，精确扣减 0.3kg（0.15kg×2 份）
- ✅ **实盘对账**：3 食材差异检测 + AI 预警
- ✅ **损耗生成**：30 食材损耗记录 + AI 分析
- ✅ **日报生成**：营收 2301 / 净利 +1190 / 毛利率 49.16% + AI 摘要建议
- ✅ **日报推送**：微信端收到日报卡片 + 洞察入库

**验证方式**：
- 按 Demo 体验指南操作，5-8 分钟跑完全流程
- 所有功能都是真实实现，不是原型/概念

## 六、已知遗留问题（如实记录，不夸大）

### 不影响 Demo 体验

1. **AI 走 fallback 模式**：未配置 TRAE_API_KEY，对话/日报摘要/进货单识别走规则匹配。结构完整，配置 key 后即启用真实大模型。
2. **dish-ingredients 路由挂载路径偏差**：BOM 查询端点实际路径为 `/api/dish-ingredients/dishes/:id/ingredients`（非标准的 `/api/dishes/:id/ingredients`）。前端未依赖此端点，不影响业务。
3. **部分食材无 active 库存批次**：seed 数据中部分食材已消耗完，下单时 BOM 扣减静默跳过。Demo 数据真实状态，不影响流程演示。
4. **损耗计算历史快照简化**：actual_consumption 用当前余量反推，存在偏差。API 正常返回，AI 分析正常触发，数值不影响 Demo 体验。

### 可提升项（如时间允许）

- 配 TRAE_API_KEY 启用真实 AI 对话和日报生成
- 损耗/对账计算改用历史快照表，数值更精准
- 三端视觉进一步提档（尤其商家端和 H5）
- 录 Demo 视频
