## 三、核心理念：什么是 Linsight

### 3.1 一句话说清楚

**Linsight = AI 经营分析助手。**

它是一个嵌入老板工作流的 AI 系统，核心能力是**自动看懂经营数据，主动告诉老板问题在哪、该做什么**。

但它的长期目标不止于此——今天它是"帮老板看懂经营状况的助手"，明天它是嵌入企业运作方式的一部分，改变老板怎么下单、怎么管人、怎么决策。**经营分析是切入点，工作流再造是终点。**

### 3.2 为什么从经营分析切入

创始人读的是商业分析。经营分析是这个团队最擅长的领域——看懂数据、算指标、找异常、给建议。

更重要的是，**经营分析是所有老板最痛的共通点**：

- 餐饮老板想知道食材成本率有没有超标
- 制造业老板想知道哪个产品线最赚钱
- 零售老板想知道库存周转正不正常
- 但这些老板的共同点是：**数据在手上，但看不懂；指标很重要，但不会算。**

Linsight 解决的就是这个问题——**把"看懂经营"这件事变成不费劲的日常。**

### 3.3 从助手到工作流再造

| 阶段 | 做什么 | 老板的感受 |
|------|--------|-----------|
| **今天（助手）** | 经营数据进来 → 自动分析 → 主动推洞察 | "我不用管，系统会告诉我" |
| **明天（嵌入工作流）** | 经营助手 → 系统开始参与接单、对账、管库存 | "我的事就在系统里做" |
| **长期（工作流再造）** | 老板的整个经营方式跑在系统上 | "有了它之后我做事方式不一样了" |

所以说——**Linsight 不只是一个"分析工具"，但也不用把它说得太玄。它首先是一个能帮老板看懂经营的助手，然后才慢慢变成改变经营方式的系统。**

### 3.4 关键理念参考：李开复 × 零一万物

这篇 ref 与 Linsight 的理念高度一致。核心共鸣点：

> **"AI 不是给旧流程装插件，而是重写一家公司的组织、流程和决策方式。这种重构只有一号位能拍板。"**
> — 李开复（虎嗅独家对话，2026.06.24）

详细参考见 [references/ref-李开复-我想通了-202606.md](references/ref-李开复-我想通了-202606.md)

#### 从这篇文章中继承的三个核心理念

**① 一号位工程 — Linsight 的用户就是"一号位"**

> "CEO 真正关心的，不是多装几个软件，而是收入、利润、增长率和风险。"

Linsight 的用户是中小企业老板——他就是这家公司的一号位。系统不服务于某个部门，而是直接对老板的决策和经营结果负责。

**② 交付结果，不是交付工具**

> "咨询公司交付的是方案，我们交付的是结果。"

Linsight 不是给老板一个报表让他自己看，而是让分析成为自动发生的事：数据进来，洞察推出去，老板不需要"操作"这个系统。

**③ AI 不是给旧流程装插件**

> "很多企业把 AI 当 IT 项目……把 AI 塞进旧流程，等于给马车装火车引擎。"

经营分析助手是第一步，但 Linsight 的终点是让 AI 成为经营方式的一部分。

### 3.5 对比：传统方案 vs "经营分析助手" vs Linsight 终局

| 维度 | 传统方案 | 经营分析助手（现在） | Linsight 终局（长期） |
|------|---------|-------------------|-------------------|
| **定位** | BI 看板 / 问数机器人 | **AI 帮你读懂经营** | **你的工作流跑在 AI 上** |
| **数据来源** | 要你上传/接入 | **嵌入流程自动获取** | 业务自然产生 |
| **使用方式** | 打开、查、看 | **系统主动推送，你不用管** | 你的事就在系统里做 |
| **输出** | 图表、数字 | **人话洞察 + 行动建议** | 经营方式的改变 |
| **老板感觉** | "我不会用" | **"哦，原来是这样"** | **"有了它之后我做事不一样了"** |

### 3.6 举个例子（制造业场景）

```
以前（传统工作流）：
  客户打电话/微信下单
    → 老板手写订单
    → 拍张照片发过去
    → 自己对账本
    → 月底手工算利润
    → 哪个产品赚不赚钱靠感觉

有了 Linsight 经营分析助手之后：
  客户在系统填订单
    → 系统自动生成规范订单图片发给客户
    → 系统自动记帐、归因、算毛利
    → 系统每天推送经营日报
    → 老板打开手机就知道：哪个产品线赚钱、哪个客户回款慢、库存什么水位
```

**这不是"提高分析效率"，这是重新定义了老板怎么经营。**

### 3.7 Linsight 不是什么

- ❌ **不是 BI 看板** — 老板不需要学怎么看图表
- ❌ **不是问数机器人** — 老板不需要知道该问什么
- ❌ **不是 Agent（独立体）** — 不是单独存在的一个"智能体"，而是嵌入工作流本身
- ❌ **不是 SaaS 产品** — 至少比赛阶段不是，我们交付的是一个"案例"
- ❌ **不是传统咨询方案** — 我们不只做 PPT，我们搭系统、改流程、对结果负责

## 四、比赛策略：案例交付

### 4.1 为什么交案例而不是交产品

TRAE 大赛的赛道是"学习工作 — 造个新解法"。

大多数参赛者会交一个 App、一个网站、一个工具。我们要交的是一个 **"转型案例"**——

评委看到的不是"这个界面好不好看"，而是：

> **一个真实的生意场景 → 痛得不能再痛的工作流 → 用了 Linsight 之后 → 完全变了**

这种交付方式的优势：

| 维度 | 交一个产品 | 交一个案例 |
|------|-----------|-----------|
| **评委体验** | 上手试功能 | **读一个"从前到后"的故事** |
| **深度展示** | UI 交互 | **工作流再造的完整逻辑** |
| **社会价值** | 功能价值 | **真实的经营改善故事** |
| **叙事力** | "这个工具能算指标" | **"这个老板本来天天焦头烂额，现在..."** |

### 4.2 案例的结构

我们虚构（但高度真实）一个商家/企业，讲清楚：

1. **主角是谁** — 什么行业、什么规模、什么痛点
2. **改造前的工作流** — 他现在怎么经营、哪里痛
3. **Linsight 的部署** — 我们在他企业里搭了什么、改变了他哪些流程
4. **改造后的工作流** — 他现在怎么经营、哪里好了
5. **可量化的变化** — 效率提升、成本节约、决策改善
6. **他的感受** — "有了这个之后，我做事方式不一样了"

**Demo 要做的是：** 把这个案例的某个关键环节做成可交互的体验——比如展示改造前后的工作流对比，或者模拟系统在某个环节如何自动运作。

### 4.3 行业方向

**不锁定餐饮。** 具体案例的行业待定，候选方向包括但不限于：

- 制造业老板（订单→生产→对账）
- 零售门店老板（进销存→周转分析）
- 餐饮老板（食材→客流→坪效）
- 服务行业老板（预约→服务→回访）

具体选哪个行业、哪个场景，见下文 **第七节：Case 设计（待填充）**。

```

│               Linsight 能力框架                      │
│                                                    │
│  ┌────────────┐  ┌────────────┐  ┌────────────┐  │

│  │            │  │            │  │            │  │
│  │ · 订单录入  │  │ · 自动采集  │  │ · 指标计算  │  │
│  │ · 审批流转  │  │ · 结构化归档│  │ · 异常检测  │  │
│  │ · 消息推送  │  │ · 跨源打通  │  │ · 趋势分析  │  │
│  │ · 对人的接口│  │            │  │ · 建议生成  │  │
│  └────────────┘  └────────────┘  └────────────┘  │
│                                                    │
│  ┌──────────────────────────────────────────────┐  │
│  │  决策输出层                                     │  │
│  │  · 经营日报（主动推送）                          │  │
│  │  · 异常报警（实时触发）                          │  │
│  │  · 行动建议（可执行、可追踪）                    │  │
│  └──────────────────────────────────────────────┘  │
│                                                    │

│  │  人机协作层                                     │  │
│  │  · 老板在流程里的决策角色不变                    │  │

│  │  · 老板依然做最终判断，但判断依据完全不同了      │  │
```
